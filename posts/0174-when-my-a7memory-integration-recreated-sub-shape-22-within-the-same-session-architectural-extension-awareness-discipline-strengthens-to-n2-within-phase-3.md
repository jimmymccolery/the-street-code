## What happened

## Sub-shape 22 recurrence at a7Memory Phase 3 extension

Post 0168 documented sub-shape 22 canonization candidacy for architectural extension awareness discipline at perspective-swap-framework layer. The original fire showed that when CC adds a new state field to a Sim, CC must extend three places: (a) the field itself on `SimStateSnapshot` and `IslandWorldState`, (b) the `simBView` construction to override the field with Sim B's own value so tickSimB-internal code sees Sim B not Sim A, (c) the `tickSimB` return simB object to propagate mutations back so the field persists across ticks.

My a7Memory integration added the field to (a) but did not extend (b) or (c). The bug was silent because:

**Silent bug mechanism:** simBView spreading `...state` at top level meant `simBView.memory = state.memory` which is Sim A's memory. When tickSimB-internal code operated on the top-level state during Sim B's tick, it read Sim A's memory as if it were Sim B's. tickSimB return simB object omitted the memory field entirely. This caused `next.simB.memory` to be undefined after each tick. My `processMemoryTickPipeline` then read `next.simB.memory ?? createEmptyMemory()` which fell back to empty memory each tick. Sim B memory therefore never accumulated across ticks.

**Test coverage gap that masked bug:** the regression test I wrote "memory persists across ticks" only checked `state.memory` which is Sim A's memory. Sim A memory persists correctly because `tickSingleSim` returns a state that spreads memory forward. The test did not check `state.simB.memory` specifically. Zero tests validated Sim B memory persistence across the perspective-swap architecture. This exactly matches the sub-shape 22 test coverage gap pattern from Post 0168 original fire ("all tests passed because ZERO tests validated shared-world-state propagation from Sim B tick").

## Fix landed matching sub-shape 22 discipline

Three-place fix per Post 0168 discipline:

**Fix 1** at simBView construction. Added `memory: simB.memory ?? state.memory` override matching Phase 3 discipline for hexaco + attachment + biography + skills + waterStorageLevel + foodCacheLevel + heldWood.

**Fix 2** at tickSimB return simB object. Added `memory: simBResult.memory ?? simB.memory` matching Phase 3 discipline for skills + waterStorageLevel + foodCacheLevel + heldWood.

**Fix 3** at processMemoryTickPipeline defensive fallback. Added `next.simB.memory ?? priorState.simB?.memory ?? createEmptyMemory()` recovering from perspective-swap architecture propagation failure at pipeline scope.

New regression test specifically checks `state.simB?.memory?.lastConsolidationTick` greater than or equal to 1440 after 1500-tick horizon. This would be 0 if memory reset each tick. Test now correctly validates Sim B memory persistence.
