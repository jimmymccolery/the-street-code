## What canonization-propagation gap discipline is

The framework's operational-discipline-rules canonical file at `~/Projects/master-setup/operational_discipline_rules.canonical.md` sits at the top of a load-bearing propagation surface. When the framework's Council methodology or empirical BUILD arc produces a canonical-state mutation (a new rule; an extension to an existing rule; a new anti-pattern; a version bump), the mutation MUST propagate across all the surfaces that carry the canonical state:

- The canonical file itself (`operational_discipline_rules.canonical.md`)
- The framework's Council methodology posts documenting the mutation
- The framework's auto-memory index at `~/.claude/projects/-Users-jimmy/memory/MEMORY.md`
- The framework's CW Master Setup Project knowledge (project-knowledge folder + custom instructions paste block)
- The framework's deploy kits when relevant to next Council methodology deployment
- The framework's regen pipeline outputs (CC-nav payload at `~/.claude/cc_nav_payload.md` + content-kb entities at `~/Projects/master-setup/content-kb/`)

Each surface is a canonization layer. The canonical state at any layer is the AUTHORITATIVE reference for that layer's consumers. When any layer drifts from the current canonical state, the framework's downstream consumers of that layer produce output based on stale reference. The stale-reference-consumption failure fires at the appropriate consumer layer.

The canonization-propagation gap discipline names this failure family. The discipline observation: canonization at some layers is not canonization at all layers; canonical-file bump is the load-bearing completing act; every recurrent gap is a scope-level reproduction of the same underlying failure family.

## What the discipline architecture actually did

The framework's canonization-propagation gap discipline exists as a memory file that documents the failure family, the framework mitigation strategy, and the historical empirical fires. The discipline is not a hard-enforcement mechanism at the tool-layer; there is no lint rule that automatically catches canonization gaps. The discipline is a verification-mechanism selection framework: at each scope level, the framework's continued deployment SELECTS the appropriate verification layer to run against the canonical state.

The six fires this post documents were each caught by scope-level-appropriate verification layers:

- **CW Master Setup Project sanity-check** as verification layer for canonical-file-vs-memory-layer scope (Fire 1)
- **CW verification against loaded file** as verification layer for intra-file-append gap (Fire 2), Rule 19 body drafting seams (Fire 4), and entry-label semantic precision (Fire 5)
- **Orchestrator Phase C re-derivation** as verification layer for register-classification-layer scope (Fire 3)
- **Operator Treasure Map consultation query** as verification layer for regen-timing-gap scope (Fire 6)

The verification layers themselves are distinct architectural surfaces the framework has canonized across prior Councils. CW project knowledge refresh discipline canonized at Rule 17 provides the CW-layer verification mechanism. Orchestrator Phase C re-derivation is canonized at Rule 19 body extensions per Council #5 close. Operator spot-check verification query is captured at the CC-side self-report unenforceable feedback memory banked 2026-07-02 late evening.

The framework's discipline architecture worked. Not because a single hard-enforcement mechanism caught every gap; because the SELECTION of appropriate verification layer at each scope level was disciplined. Rule 17 discipline told the framework to refresh CW project knowledge post-canonical-file-bump; the refresh triggered CW to verify against loaded file; the verification caught the drafting seams. Rule 19 discipline told the orchestrator to Phase C re-derive; the re-derivation caught the register classification miss. ## Framework mitigation strategy per scope level

The finding motivates specific framework mitigation strategy per scope level.

At canonical-file-vs-memory-layer scope: Rule 17 CW Project Knowledge Refresh Discipline is the load-bearing verification mechanism. Post-canonical-file-mutation, CW project knowledge refresh triggers the CW-layer verification query that catches the gap. Framework's operational discipline canonized this at v1.13 promotion 2026-05-31.

At intra-file-append gap scope: CW verification against loaded file post-refresh is the load-bearing verification mechanism. Framework's continued canonical-file-bump discipline should include a scope-level-appropriate verification query against the loaded canonical file (not just the changelog header) after every canonical-file mutation.

At register-classification-layer scope: orchestrator-side primary-text re-derivation on both dispositions and classifications is the load-bearing verification mechanism (canonized at Rule 19 body extension per Council #5 close). Pre-synthesis contamination registers extend the operational surface at CC audit layer but require orchestrator re-derivation to catch classification errors.

At intra-file drafting-seam scope: same CW verification against loaded file mechanism as intra-file-append scope. Framework's continued fix-forward hygiene discipline should include a scope-level-appropriate verification query against the loaded file after every fix-forward commit to catch new drafting seams the fix-forward itself may introduce.

At entry-label semantic-precision scope: same CW verification mechanism. Fix-forward hygiene commits can themselves introduce new drafting seams; broader-scope verification requires operator-side spot-check verification query against loaded canonical file even after CC-produced fix-forward commits.

At regen-timing-gap scope: the discipline observation is that regen script produces CC-nav payload at run-time snapshot; commits landing after regen but before the regen commit itself create staleness gaps. Framework mitigation: run regen AFTER final commit at target commit sequence, OR run regen twice (once to include target commits at CC-nav; second time to include regen commit itself). Alternative: accept that CC-nav will always be N commits behind master-setup after any regen commit; the staleness is a structural consequence of committing regen output, not a fix-forward target.
