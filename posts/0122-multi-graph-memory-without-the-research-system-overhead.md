The framework I am developing has a multi-decade memory architecture (the three-layer architecture documented in post 0118 in this arc; Layer 1 raw events plus Layer 2 consolidated entries plus Layer 3 accumulator). Pure-relational tables backing the architecture would not scale to multi-decade scope at the framework's target NPC population. Multiple separate graph databases would over-engineer the architecture for indie-budget V1 deployment. The in-between is what MAGMA-style multi-graph agent memory architectures provide; the framework's adaptation of MAGMA produces typed indexes over a single underlying memory store, which captures MAGMA's separability benefits at indie-budget cost without the research-system overhead the direct MAGMA adoption would require.

MAGMA (Multi-Graph Agent Memory Architecture; arXiv:2601.03236; published 2026) is the canonical recent research paper on multi-graph agent memory architectures. The paper develops a conceptual framework for separating procedural-agent memory into multiple typed views (semantic memory; temporal memory; causal memory; entity memory) with separated graph-database backing per view; the paper benchmarks the architecture's retrieval performance and demonstrates speedups over single-store approaches at research-system scale. The framework's research arc surfaced MAGMA as a relevant precedent during the architecture-spec phase; the framework's adaptation borrows MAGMA's typing decomposition while substituting the storage approach to fit the V1 budget. This post walks through the borrowing and the substitution.

## What MAGMA contributes

MAGMA's substantive contribution is the typing decomposition for agent memory. The decomposition observes that procedural agents need different retrieval patterns from memory for different purposes:

Semantic retrieval asks "what does X mean to this character?" The retrieval looks up the character's conceptual associations with a specific concept (the player's profession; the alley's location; a specific historical period); the retrieval returns the character's specific tagged associations rather than a flat list of events involving the concept.

Temporal retrieval asks "what happened with X recently?" The retrieval looks up events involving a specific subject (the player; another NPC; a specific situation) within a specific time window; the retrieval returns the events in chronological order with their associated context.

Causal retrieval asks "why did X do Y last time?" The retrieval looks up events whose causal relationships connect to a specific outcome (the player's previous tip leading to the NPC's subsequent gratitude; the NPC's previous interaction leading to a specific behavioral pattern); the retrieval returns events with their causal-link annotations rather than as isolated events.

Entity retrieval asks "what is the relationship state between A and B?" The retrieval looks up the pair-relational summary for a specific ordered pair (the pair sentiment scalar; the recent pair-event history; the trait-relevant pair flags); the retrieval returns the summary as a single retrievable artifact rather than as a flat list of pair-related events.

One uniformly-typed memory store cannot serve all four retrieval patterns efficiently. The store has to be scanned in different orders for different retrieval patterns; the scan cost is operationally significant; the per-retrieval-pattern cost compounds across the architecture's runtime decision-scoring pipeline. MAGMA's proposed solution is to separate the store into multiple graph databases, one per view; each database is optimized for its retrieval pattern; the architecture benchmarks demonstrate speedups over single-store approaches.

## Why direct MAGMA adoption does not fit V1

The MAGMA proposal's storage architecture does not fit the framework's V1 budget envelope. Specific issues:

Multiple graph databases mean multiple persistence engines. The framework's V1 deployment is iOS; the iOS storage envelope for save-state is in the order of single-digit megabytes for iCloud-compatible deployment; the multiple-persistence-engine approach multiplies the storage-engine code-and-data footprint without delivering proportionate per-engine value. The save-file complexity grows; the save-load latency grows (save-time under two seconds is a typical user-experience expectation); the engineering complexity grows for the framework's solo-operator BUILD-side capacity.

Indie operators cannot maintain multiple separate database engines at production quality. The QA infrastructure required to keep multiple engines reliable at production scale is substantial; the framework's V1 BUILD-side capacity does not absorb the engineering work; the post-V1 work would either consolidate the engines (defeating the MAGMA-style separation benefit) or compromise on QA (producing reliability issues at production deployment). Neither outcome is operationally desirable; the V1 commitment should avoid the multi-engine approach unless the framework's BUILD-side capacity supports the engineering work, which it does not at V1.

MAGMA is a research artifact, not a production-deployment specification. The paper's contribution is to the research-system literature; the architecture is benchmarked against research-system criteria; the architecture's production-deployment characteristics are not the paper's primary scope. The framework's V1 deployment requires production-grade engineering; the production-grade work is what the framework's BUILD-side has to produce; adopting a research-artifact specification directly would import research-system assumptions that production-deployment would have to retrofit.

## The typed-index adaptation

The framework's V1 adaptation keeps MAGMA's typing decomposition (the four-view separation across semantic, temporal, causal, and entity retrieval patterns) while substituting the storage architecture. Specifically:

Single underlying memory store. The framework's three-layer memory architecture (Layer 1 raw events; Layer 2 consolidated entries; Layer 3 accumulator) is the single underlying store; the architecture's storage characteristics are documented in post 0118 in this arc; the storage cost scales with the per-character event count bounded by the consolidation pass's promotion criteria.

Typed indexes over the store, one per MAGMA-style view. The framework's implementation adds four indexes:

Semantic index. Trait-based concept tags (the character's tagged conceptual associations; example: "polite," "regular," "high-tipper" as tags applied to other characters from the perspective of a specific character); the index supports O(1) lookup by tag. The semantic index's content is generated at consolidation-pass cadence (the consolidation pass enumerates the Layer 1 events that contributed to the character's accumulated state and generates the trait-based concept tags; the tags are stored in the semantic index for fast subsequent lookup).

Temporal index. Game-time ordered (the index is keyed by game-time; the index supports O(log N) lookup by recency window or by game-year); the index points back to the underlying memory entries (Layer 1 events within the recency window; Layer 2 entries promoted from prior recency windows). The temporal index's content is updated as events enter Layer 1 (new events added to the index in chronological order) and as events promote to Layer 2 (the index's pointer to the event updates to reference the Layer 2 representation rather than the Layer 1 raw event).

Causal index. Event-to-event links via outcome-modifier tags ("because-of-X-then-Y" pointers; the index represents the causal links the framework's encoding pathway tags during event encoding). The causal index's content is generated at encoding time (the encoding pathway tags events with their causal antecedents and consequents; the tags become index entries that subsequent causal retrieval reads).

Entity index. Pair-relationship-keyed (the index is keyed by ordered-pair (A, B); the index supports O(1) lookup by pair). The entity index's content is updated at every interaction event involving the pair (the event updates the pair's sentiment scalar; the event updates the pair's recent event list; the pair summary updates at the consolidation-pass cadence per the framework's pair-summary spec).

Each index points back to the same underlying memory entries (the indexes do not duplicate the event content; the indexes hold pointers to the events in the store). The single-store-plus-multiple-indexes approach captures the per-view retrieval speedup that MAGMA's separated-stores approach delivers without the multi-store storage overhead.

Adding an event updates all four indexes atomically. The event enters Layer 1 raw memory; the temporal index registers the event at the chronological position; the semantic index updates the tag-pointer references for the tags the event contributes to; the causal index registers the event's causal-link annotations; the entity index updates the relevant pair-summary entries. The atomic update is the implementation's commitment to index consistency; the consistency is what makes the multi-view retrieval reliable.

Storage cost. The per-event pointer-overhead is bounded (a per-event pointer-overhead in the order of 50 to 100 bytes per event across the four indexes; the order-of-magnitude is what matters for the budget calculation); the total per-character index cost is in the order of approximately 10 to 50 megabytes per character at the framework's target multi-decade scope; the order-of-magnitude is operationally tractable within the iOS V1 storage envelope (the indexes plus the underlying memory store together fit within the envelope; the consolidation pass's promotion criteria keep the per-character Layer 2 storage cost bounded; the Layer 3 accumulator is constant per character).

## Pair summaries at consolidation boundaries

At the yearly consolidation pass (per the framework's specification documented in post 0118 in this arc), entity-index entries that have accumulated more than a per-pair threshold of events get summarized into pair-summary entries. The pair summary is a compressed representation of the accumulated pair history; the summary's content is bounded (a fixed-size representation regardless of the accumulated event count); the summary's generation algorithm is deterministic (the same input event sequence produces the same summary; the determinism is required for save-load consistency).

The original events stay in the memory store after the summary is generated. The events do not delete; the events transition from being the primary retrieval target for the entity-index to being the secondary retrieval target (the primary target becomes the pair summary; the secondary target is the individual events that the summary aggregates). The runtime decision-scoring pipeline reads the pair summary at O(1) cost rather than the original events at O(N) cost; the cost reduction is what makes the runtime decision-scoring tractable at the framework's target scale.

The pair-summary-as-runtime-access pattern has commercial precedent. Sims 4 Sentiments (documented in EA's November 2020 update notes and the broader Sims community wiki) uses pair-specific affective summary entries as the runtime-accessible representation of pair-relational state. RimWorld opinions (documented in the RimWorld wiki Social mechanics pages) uses pair-specific scalar modifiers as the runtime-accessible representation. The framework's pair-summary approach borrows the runtime-accessible pattern from these precedents and extends the pattern with the framework's broader commitments (multi-decade scope; trait-modulated pair-summary content; consolidation-pass-driven summary updates).

## A retrieval scenario walkthrough

A walkthrough of a specific runtime retrieval scenario clarifies the typed-index adaptation's operational shape. The scenario: a pinboy NPC needs to decide whether to be deferential to an incoming bowler character.

Entity-index lookup. The pinboy's decision-scoring layer queries the entity index for the (pinboy, incoming-bowler) pair; the lookup is O(1) and returns the pair summary. The pair summary contains: the accumulated sentiment scalar (between negative one and positive one); the last three to five event tags from the pair's recent interaction history (compressed representations of the most recent significant events); the trait-relevant flags (whether the pair's history includes specific events that the pinboy's trait-modulated decision-scoring needs to read).

Sentiment scalar plus traits feed into active-inference decision-scoring. The decision-scoring layer reads the pair summary's sentiment scalar and trait flags; the layer combines the inputs with the pinboy's own current accumulator state (Layer 3 valence-arousal-stress); the layer scores candidate actions (be deferential; be neutral; be casual) against the combined inputs using the framework's cached active-inference policy (the cached-policy approach is documented in post 0123 in this arc).

The pinboy's decision is rendered without touching the bulk memory store. The decision-scoring pipeline's read pattern is: pair summary (O(1)); accumulator state (O(1)); cached policy lookup (O(1)); decision rendering (O(1)). The total per-frame read cost is in the order of microseconds rather than the order of milliseconds; the cost fits comfortably within the per-frame budget; the cost scales linearly with the number of active NPCs in the player's perceptual window rather than with the per-NPC event count.

Compare to the naive alternative. The naive retrieval pattern would: scan the pinboy's Layer 1 raw events for events involving the incoming bowler (O(N) over the pinboy's per-game-month event count); aggregate the events into a per-pair summary at retrieval time (O(N) over the retrieved events); feed the aggregated summary into the decision-scoring pipeline (O(1) once the aggregation completes); render the decision (O(1)). The O(N) scan plus the O(N) aggregation breaks the per-frame budget at the framework's target scale; the naive pattern is what the typed-index adaptation avoids; the avoidance is the typed-index adaptation's substantive contribution.

## What the typed-index adaptation gives up

The typed-index adaptation does not deliver MAGMA's full benchmark retrieval performance. Separated graph databases would be faster at deep semantic queries (queries that traverse the semantic graph's structure rather than reading specific tag lookups); the framework's typed-index adaptation supports the tag-lookup pattern at O(1) cost but does not support the deeper graph-traversal queries that MAGMA's separated-stores benchmarks. The trade-off is accepted; the framework's V1 design surface does not require the deeper queries; the V1 commitment is operationally tractable.

The typed-index adaptation does not support complex graph-traversal queries beyond the framework's current commitments. Queries like "find all events causally downstream of event X across all entities" would require either backing-store schema additions or a separated-graph-database approach; the framework's V1 implementation does not include the backing-store additions; the V1 commitment supports the typed-index queries but does not support the broader graph-traversal queries. The trade-off is accepted; the framework's V1 design surface does not require the broader queries; the post-V1 work or the Steam V1.0 work might revisit the trade-off if the design surface evolves to require the broader queries.

## V1 implementation cost

The schema-level work for the typed-index adaptation is bounded. The implementation adds four indexes (semantic; temporal; causal; entity); each index has a defined data structure (the index's underlying representation; the index's update interface; the index's lookup interface); the consolidation pass's pair-summary generation is part of the broader consolidation-pass implementation work. The implementation cost estimate at the framework's planning phase was roughly 24 to 40 hours of BUILD-side work; the estimate includes the index implementations, the encoding-pathway integration that updates the indexes atomically, the consolidation-pass extension that generates pair summaries, and the test-run validation across multi-game-year scope.

The implementation cost is operationally tractable within the framework's V1 budget; the cost is what the typed-index adaptation actually costs at the implementation layer; the cost is what the framework's V1 BUILD-side absorbs as part of the broader hybrid-path implementation work.

## What this changed in my workflow

Two specific changes. First, when the framework adopts a research-paper architecture as the conceptual basis for a V1 commitment, I now explicitly enumerate the production-deployment substitutions before the implementation work begins. The discipline catches direct-adoption failure modes (where the research-paper architecture is adopted without addressing the production-deployment constraints the paper did not address); the discipline produces substitution documentation that is accountable to the constraint-driven rationale; the discipline is what makes the V1 commitment operationally tractable rather than research-prototype-tractable.

Second, when the framework adopts a multi-view or multi-component architectural pattern, I now check whether the pattern's storage and complexity overhead are within the V1 budget envelope before the adoption is ratified. The discipline catches over-engineering failure modes (where multi-component patterns are adopted at indie-budget scales where the multi-component overhead exceeds the multi-component benefit); the discipline produces architecture decisions that respect the budget constraints; the discipline is what made the typed-indexes-over-single-store adaptation visible as the appropriate response to the MAGMA pattern at V1.
