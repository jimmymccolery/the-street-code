**2026-07-06 -- Tuesday evening -- Post 0190 shipment**

Council #6 close 2026-07-04 ratified session logger emission wiring at 3-of-4 substrate convergence as FOUNDATIONAL for population-scaling observability. That ratification was architectural. The code state through Post 0189 had session log data structures + append primitives + consumer UI at the primary-backend-project's IslandSimulation.tsx -- but the sim-ai reference tick loops did NOT emit into a passed-in log at TickConfig scope, so any consumer wanting per-tick event traces at O(1) per emission had to do the emission manually outside the tick loop (as IslandSimulation.tsx did via functional appendEntry which was O(N) per tick from array-clone semantics).

Post 0190 ships end-to-end emission wiring. The mutating dispatch primitive `emitToSessionLog` operates on log.entries in place. Ring-buffer eviction fires at 5% overflow threshold amortizing to O(1) per emission (contrast with 50K-entry O(N) array clone which is untenable at 500-5000 NPC populations that Council #6 population-scaling foundation pass targets). Determinism is preserved because sessionLog is not part of the simulation state closure -- verified by side-by-side stateA (with sessionLog) vs stateB (without) tick-by-tick 30-tick empirical comparison at seed=42 showing needs + hasShelter + hasCaughtFish + layer3 all bit-identical.

### The two-Scene sibling logger pattern

Island Scene has been the reference Scene for sim-ai since Phase 1. Its sessionLogger.ts is tightly typed to IslandWorldState -- detectEventTags detects survival mechanics (shelter_built + storm_start/end + day_change + fish_caught + water_scavenged). The domain-specific Scene (bowling-tycoon domain) has entirely different semantic (non-lethal + shift-time structure + pit-work primary loop + coworker mentorship + injury landmark). Rather than force IslandWorldState-shaped state through both Scenes (which would violate the Scene abstraction refactor completed 2026-07-04), Post 0190 ships a sibling `bowlingAlleySessionLogger.ts` scoped to the domain-specific Scene's event surface. The pattern is symmetric: BowlingAlleySessionLog type + createBowlingAlleySessionLog factory + emitBowlingAlleyEntry mutating primitive + emitBowlingAlleyTickEvents dispatcher + detectBowlingAlleyEventTags detector + summarizeBowlingAlleySession consumer surface.

This establishes the sibling-logger-per-Scene pattern as canonical. When a companion life-builder Scene ships or a Rooms Scene ships, they follow the same shape: Scene-scoped event tag union + Scene-scoped log type + mutating emit primitives + tick loop wiring via optional sessionLog on TickConfig.

### The cross-product aggregation surface

Individual Scene loggers solve per-Scene observability but do not solve cross-product observability. The ecosystem's canonical entity registry has always been the cross-product tracking surface for entities. Post 0190 ships `crossProductTraceAggregator.ts` at `sim-ai/src/room-to-life-universe/` operationalizing cross-Scene trace collection. The key design constraint: aggregator is Scene-agnostic. Adding a new Scene type requires only adding a new discriminant variant to TraceEntry, not modifying export or summary code paths.

Consumers register traces via addIslandTrace or addBowlingAlleyTrace with (label, product, log) triples. ProductScope enum covers `bowling_tycoon_simulation` + `rooms` + `companion_life_builder_product` + `research_surface` + `unspecified` -- the whole publisher-entity ecosystem plus a research surface plus a catch-all. summarizeCrossProductTrace iterates entries dispatching per-scene to the appropriate summarize function and returns a per-product breakdown + per-entry TraceEntrySummary as discriminated union so consumers narrow safely via `if (entry.scene === 'island')`. buildCrossProductTraceExport produces the canonical export shape: version + createdAt + exportedAt + summary + logs array with full per-entry payloads for downstream replay.

### The primary-backend-project React state migration

IslandSimulation.tsx at the primary-backend-project had been using the pre-Post-0190 pattern: session log lives in React state, tick loop functional-appendEntries into a new log object per tick, React re-renders on state change. At high tick counts + 30fps this becomes expensive: 100K-entry array clone at 30fps is 3M array-element copies per second, dominated by the O(N) clone.

Post 0190 refactor: session log lives in useRef (mutated in place); logVersion counter increments each tick to trigger re-render of summary + entry-count + export-button-state UI. Tick loop passes sessionLog: sessionLogRef.current as TickConfig.sessionLog -- tickIslandWorld internally calls emitTickEvents which mutates log.entries in place. React's Object.is comparison for sessionLogRef.current stays stable (same ref) so React knows nothing changed -- but logVersion state change forces the re-render + summary useMemo recomputes with fresh sessionLog.entries.length read.

This pattern is canonical for high-frequency mutating updates in React: useRef for the mutating target + version counter for re-render triggering + useMemo with [version] dep array for derived summary state.
