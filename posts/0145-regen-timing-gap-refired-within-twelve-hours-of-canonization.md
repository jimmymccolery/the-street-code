## The regen-timing-gap as canonized

The framework maintains an auto-generated spatial-topology payload that documents the structural state of the operator's project workspace. The payload is regenerated by a Python co-production pipeline that runs after canonical file mutations to keep the payload's per-project sections current. The payload is authoritative for spatial-topology queries and is loaded into fresh CC sessions as a first-class context surface.

The pipeline's output includes the payload itself plus updated content-kb entities plus a Word-format operator export. When the pipeline runs, it generates the payload against the current state of each tracked project at that moment. The pipeline's own commit is then created after the payload has been written. This produces a specific temporal ordering: payload generation occurs at time T; the payload records each project's most recent commit as of time T; the pipeline's commit itself is created at time T+delta.

The consequence is that the payload records each project's state as it existed at time T. The regen commit at time T+delta is not reflected in the payload because the payload was already frozen before the regen commit came into existence. The payload is stale by exactly one commit at the master-setup zone (the zone where the regen commit lands) until the next regen runs and captures the prior regen commit as history.

Post 0143 named this sub-shape the regen-timing-gap and canonized it as entry #6 in the enumeration of canonization-propagation gap sub-shapes. The framework's mitigation strategy noted in the canonization was direct: run the regen script AFTER the final commit at a target sequence, or run the regen script twice in succession so the second regen captures the first regen's commit.

The re-fire is not a bug in the pipeline. The pipeline behaves correctly; the pipeline records each project's state at pipeline-run time, then commits after the recording is complete.
