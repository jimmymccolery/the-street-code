This is what happens after that lesson is named, the recovery path is chosen, and the session continues to its actual conclusion. The framework has a Phase 5 verification report at a fixed filepath. When subsequent Phase 5 cycles run on the same canonical-state-advancement work, which happens whenever scope expansion drives a Phase 3 rewrite, the prior Phase 5 report gets overwritten at the same path. The deployed canonical state now references a verification record that no longer exists.

This post extends [the Post 08 lesson](0008-when-deployment-outpaces-decisions-a-mid-iteration-alignment-problem.md) rather than re-deriving it. Post 08 was about Phase 3 deployment timing. Post 08's pattern surfaced through internal verification: Phase 5 caught the alignment problem on the rewritten Phase 3, which is exactly what Phase 5 is for. ## The first-order lesson

For readers who haven't read Post 08: my architecture-revision cycle runs in three substrate-crossing phases. Phase 1 reads canonical files in Claude Code and produces a verified-content document containing every canonical claim the cycle's writeup will need. Phase 3 runs in Claude on web; it synthesizes the writeup, making decisions about anti-pattern promotions, refinement evaluations, and version bumps. Phase 5 runs back in Claude Code; it verifies Phase 3's proposed canonical bytes against canonical files independently. Each phase happens on a different substrate execution; the substrate-crossing is the structure the framework's independence-as-reliability claim rests on.

The cycle's contract works cleanly when the scope plan is stable before Phase 3 runs. The architecture's failure-mode prevention assumes Phase 3 has settled inputs.

Post 08 named what happens when that assumption fails. In the cycle I ran during S006, I deployed canonical state from an intermediate Phase 3 output. The Phase 3 was real work, produced against the scope plan as it existed at that moment. I deployed the changes. The cycle felt like it had reached closure. Then the session continued: the scope plan got corrected (additional earned lessons surfaced; an existing lesson's mechanism description was rewritten when I noticed an attribution error; a second refinement candidate was added). I produced a rewritten Phase 3 against the corrected scope. Phase 5 verified the rewritten Phase 3 cleanly but caught the alignment artifact: the rewritten Phase 3's proposed bytes diverge from the canonical bytes I'd already deployed earlier in the session.

The architecture's contract was honored on both Phase 3 outputs. The alignment problem isn't an architecture defect. It's a session-management lesson: architecture-revision cycles should deploy canonical state only after the full scope plan is stable AND Phase 5 verification is complete on the final Phase 3 output.

Post 08 named three operator-side recovery paths once the alignment problem surfaces: Path A (accept the deployed state, record additional content elsewhere); Path B (roll back canonical and redeploy from final output); Path C (treat rewritten output as superseded). I chose Path A. The deployed canonical state was real production record I didn't want to disrupt. The additional content from the corrections went into the verification report and scope plan rather than the deployed Session Log row. I named the rule. The post shipped.

That's where Post 08 ended. This post starts at what happened after Path A was chosen and the session continued to its actual conclusion.

## The continuation

After Path A, the session produced the rewritten Phase 3 reflecting the corrected scope. A new Phase 5 ran on the rewritten Phase 3. The Phase 5 invocation prompt explicitly instructed it to overwrite the prior Phase 5 report at the same canonical filepath. The convention has always been one Phase 5 report per cycle, single canonical record per session. The instruction encoded that convention: produce the verification report, save it at the canonical filepath, replace whatever was there before.

That seemed obviously right at the time. The framework's whole point is canonical singularity. Multiple Phase 5 reports at the same path would create exactly the kind of canonical-state ambiguity the framework's discipline is designed to prevent. The rewritten Phase 5 was the verification record for the cycle's final Phase 3 output; it should be the canonical record of the cycle's verification work. The prior Phase 5 (produced against the intermediate Phase 3, before the scope expansion and rewrite) was now superseded by the rewrite. Overwriting it preserved the canonical-record-per-cycle convention.

The structural consequence wasn't visible at the time.

The deployed canonical state from Path A (the new anti-pattern rows #22 and #23, the Session Log row 006, the four earned-lesson entries) had been produced from the prior Phase 5's findings. The prior Phase 5 had verified the intermediate Phase 3 and produced specific findings about that Phase 3's accuracy and outstanding QUESTIONABLEs. Those findings drove the language in the deployed Session Log row, specifically the row's characterization of one of the verification findings. The row 006 entry I deployed said "1 minor labeling-discipline QUESTIONABLE on verbatim-vs-summarized attribution corrected pre-deploy."

Once the Phase 5 v2 rewrite overwrote the Phase 5 v1 file, that prior verification record stopped existing at any path. The deployed canonical state (the row 006 text I'd written from then-current operator understanding of what the prior Phase 5 had caught) could no longer be cross-checked against any extant primary record. The understanding lived in operator memory and in the deployed canonical text; the primary record that produced the understanding was gone.

The PRE-EDIT_ARCHIVED Phase 3 still exists. A reasonable question is whether re-running Phase 5 against that archived Phase 3 would reproduce the prior verification record. It wouldn't. Phase 5's verification baseline is the canonical state at the time of its run: anti-patterns, lessons-learned, Session Log, scope plan. The canonical state has advanced since Phase 5 v1 ran: anti-pattern rows v1.2 → v1.3 are now deployed; row 006 is on the Session Log; the cycle's lessons are in the lessons-learned record. A re-run would verify the archived Phase 3 against the current canonical state, not against the state v1 verified against. The findings would differ because the verification inputs differ. The prior Phase 5's specific verification work was contingent on the canonical state at the time it ran; once that state advanced through deployment, the run became non-reproducible: not just file-absent, but informationally lost.

I closed the cycle. I shipped Post 08 the next day. I moved on to the brochure rewrite work that became this post's second cluster. I did not at the time recognize that the Phase 5 overwrite had created a structural condition the framework's normal verification pass couldn't detect. The cycle's own Phase 5 had verified clean. The deployed canonical state matched my operator memory of what had been verified. There was nothing that any single-substrate review would have flagged.

The pattern's specific shape: deploy canonical changes from one Phase 5 report; overwrite that Phase 5 report with a subsequent cycle's report; the deployed canonical state now references a verification record that no longer exists at any extant location. The drift between deployed canonical state and canonical primary record is invisible until something forces a cross-check. Operator memory holds the understanding that produced the deployed text. As long as the deployment isn't questioned, the drift sits silently.

That's the structural condition. The next section covers what forced the cross-check.

## The audit that surfaced it

Two days after the cycle closed, I submitted my framework's brochure to Gemini Deep Thinking for cross-substrate audit. The brochure was a separate piece of work: a three-page articulation of what the framework is, intended for readers who haven't followed the arc closely. Cross-substrate audit on substantive Substack writeups is an established pattern in my workflow. Gemini gets the artifact cold; the audit produces grading across multiple dimensions, surfacing of generative material, and specific empirical claim verification.

Gemini's audit flagged a list of empirical claims as worth verification. One claim on the list: the brochure's reference to "validated... with 23/23 PASS and 32/32 PASS verification results respectively" for the architecture's two work classes. Gemini's reasoning, paraphrased: perfect pass rates are statistically anomalous in probabilistic LLM generation; this often indicates a permissive verification gate or reporting only post-recovery metrics rather than first-pass accuracy. The flag wasn't an accusation; it was a request to verify against canonical state.

Cycle 1 verified clean against canonical state: 23/23 first-pass; Gemini's flag on Cycle 1 didn't surface anything.

I addressed the flag by reading the Phase 5 report for Cycle 2 directly (the architecture-revision work class validation, which is the cycle this post is built around). The report showed 33 total claims, 32 PASS, 0 FAIL, 1 QUESTIONABLE. The QUESTIONABLE was C7, on canonical-alignment for proposed earned lessons against deployed lessons-learned state. The Phase 5 report itself framed C7 as "workflow-ordering, not Phase 3 defect." That's the alignment problem Post 08 named.

The brochure's "32/32 PASS" framing was technically accurate (32 of 33 claims passed) but elided the QUESTIONABLE. Gemini's instinct to flag was directionally right: the brochure's framing was simplified in a direction that under-stated what the verification had caught. That's a Tier 1 brochure correction worth applying.

Then I cross-checked against my own Session Log row for the same cycle. Row 006's text, deployed during the cycle: "Phase 5 verification VALIDATED architecture for architecture-revision work class (32/32 PASS, 1 minor labeling-discipline QUESTIONABLE on verbatim-vs-summarized attribution corrected pre-deploy, zero substance-fabrication fires, zero premature-promotion fires, refinement-decision and version-bump consistency clean)."

The row says "labeling-discipline QUESTIONABLE on verbatim-vs-summarized attribution." The current Phase 5 report says C7 / canonical-alignment / workflow-ordering with explicit zero verbatim-mislabeling fires. Two canonical records of the same cycle's verification disagreed on what the QUESTIONABLE actually was.

I also checked an adjacent canonical artifact, my system-schema state inventory file, which captures Cycle 2 verification status as a row in a snapshot table. The state inventory said the same thing as row 006: "labeling-discipline QUESTIONABLE corrected pre-deploy." Two downstream canonical-record summaries diverged from the canonical primary record (the Phase 5 report itself) in identical language.

The discrepancy mattered for two reasons. One: same-substrate pre-ship verification on the brochure had not caught either Tier 1 contradiction the audit flagged (project count arithmetic mismatch; Substack timing inconsistency between Page 2 and Page 3). Cross-substrate audit caught what same-substrate verification had missed. Two: the row 006 / state-inventory drift was a separate finding from what Gemini had asked about. Gemini flagged the figure; verifying the figure surfaced the drift; the drift was its own canonical-state question. Without the cross-substrate audit prompting the verification, neither finding would have come to light.

That's what made me start the reconstruction.

## The reconstruction

The investigation question: which canonical record characterizes the Cycle 2 QUESTIONABLE accurately, row 006 / state-inventory ("labeling-discipline corrected pre-deploy") or the Phase 5 report itself (C7 / canonical-alignment / workflow-ordering)? And how did the divergence get there?

I started with the canonical Phase 5 report and read it in full. The report is unambiguous: 32 PASS, 1 QUESTIONABLE = C7 on canonical-alignment for proposed lessons 5-6 vs deployed lessons-learned state. The report explicitly notes verbatim-mislabeling fires equal zero. The report's own framing of the QUESTIONABLE: "workflow-ordering, not Phase 3 defect." The labeling-discipline framing isn't anywhere in the report.

Then the report's meta-finding section gave me what I'd been missing. The current canonical Phase 5 report wasn't the only Phase 5 report that had existed at that filepath during the cycle. A prior Phase 5 report (30,981 bytes, dated 2026-05-03 21:46) had existed at the same path. That prior report had verified an earlier Phase 3 (which still exists in the experiment directory under a `_PRE-EDIT_ARCHIVED` filename suffix) against an earlier scope plan (4 earned lessons, 1 refinement candidate, before the in-cycle expansion). The current Phase 5 report (34,192 bytes, dated 2026-05-04 11:12) was produced against the rewritten Phase 3 and the corrected scope, and it explicitly overwrote the prior Phase 5 at the same path per operator instruction in the invocation prompt.

The prior Phase 5 report is unrecoverable from any canonical location. The PRE-EDIT_ARCHIVED Phase 3 still exists; the prior Phase 5 doesn't.

That changed the question. The deployed row 006 was written from the prior Phase 5's findings. I deployed canonical state when the prior Phase 5 was the verification record. The "labeling-discipline corrected pre-deploy" framing was what I'd written from operator memory of what the prior Phase 5 had reported. Whether row 006 accurately captures the prior Phase 5's actual findings, or whether I'd mischaracterized them in the deployed text, isn't determinable from extant canonical state. The prior Phase 5 is gone.

I considered three possibilities. Possibility A: row 006 accurately captures the prior Phase 5's findings, which were distinct from the rewritten Phase 5's findings (the prior had a labeling-discipline QUESTIONABLE that the rewrite resolved or relabeled). Possibility B: row 006 mischaracterized the prior Phase 5's findings, and the actual finding was always the C7 / canonical-alignment item the rewrite documents. Possibility C: insufficient evidence to distinguish A from B.

The current Phase 5 report's contents lean against A. The current report's verification language is internally coherent: zero verbatim-mislabeling fires, the QUESTIONABLE is C7 specifically. If the prior Phase 5 had a labeling-discipline QUESTIONABLE that got resolved, the current report would more likely name the resolution. It doesn't.

But the prior Phase 5 doesn't exist. I can't read it. I can't quote it. I can't compare its findings to row 006's text. The verdict that holds, honestly, is Possibility C with a lean toward B. The current canonical Phase 5 record doesn't support row 006's framing. The prior Phase 5 might have, and that prior record is unrecoverable.

Reconciliation between row 006 / state-inventory and the canonical Phase 5 report is operator-side decision territory at the next architecture-revision session. Three reconciliation options exist: update row 006 to match the current Phase 5 (treat row 006 as having mischaracterized); preserve row 006's framing if operator memory or other side-channel evidence corroborates a real prior-Phase-5 distinct finding; split the row to capture both findings if both substantiate. None of these is forced by tonight's reconstruction. The reconstruction surfaces the verdict; correction work is properly deferred.

## The pattern named

Once the reconstruction landed, the workflow-ordering pattern naming was straightforward. I filed it in my research queue as Topic 18: intermediate Phase 5 reports overwritten by later cycles leave deployed canonical state referencing unrecoverable verification.

Single fire on the specific Phase-5-overwriting mechanism (n=1, the cycle this post documents). Below the multi-fire bar for canonical anti-pattern promotion, which my framework requires before discipline changes get encoded as canonical rules. Structural pattern with clear recurrence conditions: any architecture-revision cycle where scope expands mid-cycle and Phase 3 rewrites can produce another fire of this exact mechanism. Whether that specific recurrence happens is empirical question for future cycles.

But the family pattern this fire belongs to has more empirical surface area than the single-mechanism fire suggests. While drafting Topic 18 itself, I produced two cross-reference path drifts: I named state_inventory_2026-05-05_HISTORICAL.md as living in master-setup/historical/, when it actually lives in system-schema/historical/; I named the PRE-EDIT_ARCHIVED Phase 3 with a prefix-style filename pattern, when the actual filename uses PRE-EDIT_ARCHIVED as a suffix. Both drifts were caught during verification of Topic 18's cross-references, before the writeup proceeded. Different mechanism than Phase 5 overwriting: these are memory-vs-canonical-state drift during summary drafting, not file overwriting. Same family of canonical-state-drift in canonical-state records, different operational level.

And the lesson count drift surfaced separately. Row 006 says "Four earned lessons drained to lessons-learned.md (none promoted to canonical anti-pattern; insufficient empirical evidence per S005 multi-fire bar)." The canonical primary record (lessons-learned.md) actually has seven in-cycle 2026-05-04 entries by end of cycle: the initial four that match row 006's count, plus two additional lessons that surfaced during the in-cycle scope expansion (the same expansion Post 08's first-order lesson named), plus one additional lesson the cycle's own Phase 5 caught (the workflow-ordering lesson itself). Three in-cycle lessons are missing from row 006's count of four. The drift is three in-cycle lessons, not five. (Two additional 2026-05-04 entries exist with explicit `(deploy)` heading suffix: deploy-discovered work, methodologically distinct from in-cycle drift.)

The in-cycle lesson at lessons-learned line 572 is itself the primary-source attestation of this drift. The entry describes the rewritten Phase 3's proposed bytes diverging from already-deployed canonical bytes in three places, listing "earned-lesson count" as one of three diverging artifacts. The canonical record at line 572 names the count drift the cycle's own Phase 5 caught.

Three fires of the family pattern in one investigation period, but they're not the same kind of failure. The file-overwriting fire is a deterministic architectural failure: the Phase 5 invocation prompt explicitly instructed overwriting, so under the same conditions (scope expansion mid-cycle plus Phase 3 rewrite plus the same invocation prompt) the failure recurs 100% of the time. The drafting-drift fire and the lesson count drift fire are probabilistic operator-discipline failures: they happen when downstream summaries get written from memory rather than from canonical filesystem read, and they recur stochastically when conditions are similar but not deterministically. All three sit in the canonical-state-drift family pattern Anti-pattern #10 covers (canonical-source discipline; read the canonical file when the question is about it). They operate at different levels: between Phase 5 reports and deployed canonical state; between summary drafts and canonical filesystem state; between Session Log rows and the canonical primary records they summarize. Same discipline applied at different levels. But the deterministic versus probabilistic distinction matters for what the family-pattern observation can do, which is what the next section is about.

## The refinement consideration

The mitigation is structurally clear: archive Phase 5 reports rather than overwriting them when subsequent cycles run on the same canonical-state-advancement work. Concretely, this would mean Phase 5 v1 saves as `phase_5_verification_report_S006_2026-05-04_v1.md`; if scope expansion warrants a Phase 5 v2, it saves as `phase_5_verification_report_S006_2026-05-04_v2.md` rather than overwriting v1. Final canonical references (Session Log rows, brochure references, lessons-learned attributions) explicitly cite which version produced the deployed canonical changes.

Cost/benefit analysis: costs are minor and one-time. One additional file per Phase 5 cycle when re-runs occur (negligible disk usage). Slight convention overhead in Phase 5 invocation prompts. A one-time learning curve for me to remember to version rather than overwrite. Operational file updates to drain prompts and the bootstrap manifest. Benefits are structural and ongoing. Deployed canonical state can always be corroborated against the specific Phase 5 report that produced it. Drift like the row 006 mismatch becomes structurally impossible because the v1 Phase 5 still exists for verification. Methodology research material accumulates across versioned reports: comparing v1 vs v2 reveals what changed in scope expansion plus Phase 3 rewrite, which is itself learning material. The convention parallels the existing PRE-EDIT_ARCHIVED Phase 3 convention, which already preserves Phase 3 predecessors through naming.

The empirical-bar question is harder. My framework has an explicit multi-fire bar for canonical anti-pattern promotion: a pattern needs at least two independent empirical fires before it gets promoted from operator-side discipline to canonical rule. The bar's purpose is to protect against over-correction on patterns that might not recur: single-fire patterns can be one-off coincidences, not real structural failures. Topic 18's specific Phase-5-overwriting mechanism has fired once. The bar would normally hold the discipline change for a second fire.

But the bar's reasoning was developed for probabilistic patterns. A pattern that fires once might genuinely not recur because the conditions that produced it were idiosyncratic. The Phase 5 overwriting mechanism is different in kind. The invocation prompt instructing overwriting isn't operator one-off discipline. It's part of the framework's encoded operational architecture. The drain prompts, bootstrap manifest, and standard cycle pattern all assume one Phase 5 report per cycle at fixed canonical filepath. Operator following framework convention reproduces the failure deterministically under the same conditions, because the convention itself is the mechanism producing the failure.

The distinction has limits. For a personal framework where one operator built and operates the framework, separating framework-as-architecture from operator-as-discipline is fuzzier than in multi-operator systems. But the recurrence pattern matters: framework-encoded failures recur whenever encoded conditions hold; operator-discipline failures depend on attention calibration in specific moments. The empirical-bar reasoning developed for probabilistic patterns doesn't transfer cleanly to deterministic mechanisms even with the fuzziness acknowledged.

The distinction isn't new vocabulary. Anti-patterns #22 (substrate-binding gates require structural enforcement) and #23 (hosted chat interfaces cannot support structural orchestration) are framework-encoded deterministic failures, both promoted in S006 itself. Anti-pattern #5 and the drafting-drift family are probabilistic operator-discipline failures. Topic 18's Phase 5 overwriting mechanism sits in #22 and #23's architectural-failure-mode neighborhood, not in #5's discipline-failure neighborhood. The deterministic-versus-probabilistic distinction is a structural feature already implicit in the framework's anti-pattern taxonomy; the empirical-bar question makes it explicit.

That distinction sharpens the empirical-bar question. Three positions remain reasonable, but they're now positions about whether the multi-fire bar should apply uniformly across all pattern types or whether it should distinguish deterministic flaws from probabilistic patterns.

Position A: maintain the multi-fire bar uniformly across all pattern types. Conservative; protects against over-correction; treats the bar's structural intent as the load-bearing principle even when specific cases (deterministic flaws) might not need that protection. The cost is continued exposure to the deterministic flaw's drift accumulation between now and the second fire.

Position B: discipline change at n=1 for deterministic architectural flaws specifically, flaws where the structural mechanism guarantees recurrence under same conditions. Preserve the multi-fire bar for probabilistic patterns where the bar's protection is load-bearing. The asymmetric reasoning: cost of waiting on a deterministic flaw (continued drift accumulation under guaranteed-recurrence conditions) exceeds cost of premature change (minor disk usage, convention overhead). The reasoning doesn't generalize to probabilistic patterns where "might not recur" is real.

Position C: defer the meta-discipline question itself. Whether the deterministic versus probabilistic distinction should affect bar application is its own decision worth deliberation, separate from the specific Phase 5 versioning decision.

The family-pattern multi-fire (three mechanisms in one investigation period: file overwriting, drafting-drift, in-cycle lesson count drift) is methodology research observation about the canonical-state-drift discipline's surface area. It does not license treating the deterministic Phase-5-overwriting mechanism as multi-fire validated by way of probabilistic drift fires that don't share the same recurrence conditions. The family-pattern observation surfaces a sharper methodology research question (whether the multi-fire bar's reasoning applies uniformly across deterministic and probabilistic mechanisms, or whether it should distinguish between them), but that question is operator deliberation territory at the next architecture-revision session.

The decision is deferred. None of the three positions is forced by tonight's reconstruction. Tonight the pattern is named, the deterministic-versus-probabilistic distinction is surfaced, and the refinement consideration is captured; the operator decision happens at fresh attention against the broader update bundle queued for the next architecture-revision session.

## What this demonstrates about cross-substrate audit

Two cross-substrate audit instances in 48 hours produced specific value the same-substrate verification chain hadn't produced. Gemini's brochure audit caught two internal contradictions in the brochure (a project count arithmetic mismatch; a Substack timing inconsistency between Page 2 and Page 3) that my own pre-ship verification (running on the same substrate as the drafting work) had not caught. Then verification of Gemini's flagged figures led to the row 006 reconstruction work that surfaced the canonical-state-drift this post documents. Both became visible only because a different substrate read the work cold: the first as a direct catch by Gemini's audit, the second as operator reconstruction triggered by Gemini's flag on a different empirical claim.

The mechanism is independence of verification gates. My framework's three-phase architecture already encodes this principle within a single cycle: Phase 1 reads canonical files in Claude Code; Phase 3 synthesizes in Claude on web without Phase 1 doing the synthesis; Phase 5 verifies in Claude Code without Phase 3 doing the verification. Independence is the reliability mechanism, not just thoroughness or operator vigilance. Cross-substrate audit extends the same principle one level out: a different substrate runs on the work the first substrate produced and verified. The reliability mechanism is the same: independent reads catch what coupled reads miss.

This mirrors academic peer review structure (author self-review, peer reviewers, editor judgment, post-publication critique) compressed to single working sessions. Peer review's reliability comes from the independence between gates, not from peer reviewers being smarter than authors. The same structural principle applies at the cross-substrate audit level: the audit catches what same-substrate review can't, not because the audit is smarter, but because it's independent.

There's a specific empirical observation about when cross-substrate audit produces value. The work my pre-ship verification had already passed was where Gemini's audit found contradictions. The Phase 5 report Cycle 2 had verified clean was where reconstruction found canonical-state-drift. Cross-substrate audit's value compounds when it runs against work that has already passed single-substrate review. That's the work where the same-substrate gates would have caught what they could catch, and what remains is what same-substrate gates structurally can't catch. Running cross-substrate audit as a substitute for first-pass review wouldn't produce the same compounding effect, because cross-substrate audit isn't substituting for verification work the same substrate could do; it's adding what the same substrate can't do.

The pattern this post documents would have remained undetected indefinitely without external prompting. Same-substrate verification can't catch its own stale references: Phase 5 v2 verifies the rewritten Phase 3, not the deployed canonical state from Phase 5 v1's findings. The drift in row 006 lives at a level the cycle's own verification chain doesn't reach. Cross-substrate audit reached it, and only because Gemini happened to flag adjacent figures that prompted the verification.

## What I'm not claiming

Phase 5 versioning isn't universally right. The cost/benefit analysis favors adoption, but the operator decision at the next architecture-revision session is genuine: Position A (uniform multi-fire bar) is reasonable; Position B (discipline change at n=1 for deterministic architectural flaws specifically) is reasonable; Position C (defer the meta-question of whether deterministic versus probabilistic should affect bar application) is reasonable. The decision pends fresh attention, not tonight's reasoning.

Cross-substrate audit doesn't catch everything. It caught these specific failure modes; other failure modes exist that even cross-substrate audit misses. The empirical case is two catches in 48 hours, both of specific kinds. Generalization beyond the specific catches isn't supported by what tonight's work surfaced.

n=1 promotion isn't the right call for any pattern type. The framework's multi-fire bar exists for good reasons: probabilistic single-fire patterns sometimes don't recur, and discipline changes encoded prematurely accrete framework friction without preventing recurrent failure. The deterministic-versus-probabilistic distinction sharpens the question (whether the bar's reasoning applies uniformly to both pattern types) without resolving it. The three-position deliberation is genuine and unresolved.

The workflow-ordering pattern isn't novel as a class. Canonical-state-drift in canonical-state records probably surfaces in any iterative work where outputs drive state changes: Git workflows where branches merge before review comments resolve, document workflows where shared edits propagate before all reviewers respond, research workflows where conclusions get cited before analysis is finalized. What's distinctive about this instance is the mechanism specificity (Phase 5 report persistence after subsequent cycles) rather than the existence of the broader phenomenon.

I'm not claiming canonical state can fully supersede operator memory. The framework's premise is that canonical state is the authoritative record and operator memory should defer to it. The reconstruction surfaced a space where the framework had to fall back on operator memory: the row 006 text I'd deployed was written from my memory of what the prior Phase 5 had reported; the prior Phase 5 was overwritten; operator memory is now the only bridge between deployed canonical text and the verification record that produced it. The reconstruction work documenting this drift is itself bridged by operator memory: what I remember about the prior Phase 5's findings, the cycle's scope expansion, the deployment sequence. The framework's pattern is firing in the act of documenting it. This isn't a defeater for the reconstruction; it's a structural condition the reconstruction operates under. When the canonical primary record is unrecoverable, the canonical downstream record's accuracy depends on operator memory, which the framework's discipline otherwise treats as unreliable. The framework can't fully eliminate this fragility without preserving every primary record indefinitely; it can mitigate by reducing situations where canonical text gets written from memory rather than from extant primary records, which is part of why the Phase 5 versioning consideration matters.

The empirical case is one investigation period, three related fires, one mechanism with a structurally clear mitigation. The pattern is real but the extent of its generalization is open.

## What I am claiming

Workflow-ordering at the Phase 5 report persistence level is a real pattern, distinct from Post 08's first-order lesson on Phase 3 deployment timing. They operate at different operational levels. Post 08's pattern surfaces through the cycle's own verification chain (Phase 5 catches the alignment problem on the rewritten Phase 3). This pattern surfaces only through cross-substrate audit prompting reconstruction work the cycle's own chain doesn't reach.
