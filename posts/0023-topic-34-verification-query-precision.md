Verification chains operate against canonical state. The structural assumption is that when a verification query is asked, both the substrate executing the query and the operator routing the answer are testing against what canonical state literally specifies. S009 surfaced a class of fire where that assumption fails: the verification query itself was drafted against an inferred-or-composite role taxonomy rather than against the literal canonical schema, producing queries that tested for things canonical state doesn't actually specify and would correctly fail to satisfy.

The empirical record is small: n=2 fires within the same S009 corrective sub-stage 2 arc. The first fire surfaced when Gemini Deep Think (third leg of Council of Three triangulation; covered in [Post 20](0020-council-of-three-third-leg-fallibility.md)) caught a verification-query drafting error across three substrates simultaneously. The second fire surfaced after the first fire's remediation, when Gemini Audit A re-run Dim 1 caught partial-absorption fingerprint at the persona-section text propagation layer. Both fires are sub-classes of anti-pattern #10 (the canonical-source discipline that requires reading canonical record when the question is about canonical record); both are filed in canonical record as Topic 34 (research_queue.md lines 1340-1365).

This post documents the verification-query precision class as canonized in research_queue.md, the n=2 empirical fires that surfaced its two distinct fire shapes, the partial-absorption fingerprint extension that established the second shape as a second-fire variant within the class, and the relationship between this class and the canonical-source-discipline family that anti-pattern #10 governs.

## The pattern named

A verification-query precision fire happens when a query drafted to verify canonical state is itself drafted from session-memory of canonical state rather than from a literal canonical-schema read. The error class, canonized verbatim at research_queue.md line 1344: "operator-facing verification queries drafted against an inferred or composite role enumeration rather than against the literal canonical role architecture. Drafting from memory of the rename rather than from the canonical schema produces queries that test against a role taxonomy that does not match deployed canonical state."

The mechanism is pattern-completion bias under salience pressure. When a recent canonical-state mutation makes multiple role names salient simultaneously, drafting against session-memory of the architecture leaks role names into a list shape that visually parallels the canonical structure but doesn't match it literally. The salience-drift sub-pattern, canonized at research_queue.md line 1348: "All 6 Beatles names (4 standing + 2 embedded) are salient post-rename, but only 4 are standing-role canonical references. When drafting under salience pressure (recent rename event, pattern-completion bias toward enumerating all 6 names), Claude's enumeration leaks the 2 embedded names into a list shape that visually parallels the canonical 4-standing list. Verification-query drafting against the canonical schema (literal read of role_output_schemas.md or Master_Custom_Instructions.md ROLES section) prevents this leak; drafting from session-memory of the rename produces it."

The remediation rule, preliminary pending architecture-revision canonization, at research_queue.md line 1350: verification-query drafting against canonical role architecture must read the literal canonical schema (role_output_schemas.md or Master_Custom_Instructions.md ROLES section) before drafting the query, not paraphrase from session-memory of the rename. Topic 34 is a sub-class of anti-pattern #10, generalizing the canonical-source-discipline rule from substantive claims to operator-facing query drafting precision.

What separates this class from substantive verification-content errors: the precision class operates at the question-asking layer, not at the answer-content layer. A substantive error produces a wrong answer to a correctly-asked question. A precision-class error produces a question whose answer cannot match canonical state regardless of the answering substrate's reading discipline. Audit A's frame at research_queue.md line 1346 captures this directly: "creates a verification task that is impossible to pass based on the current-state body content."

## The sub-pattern: partial-absorption fingerprint as second-fire shape

The two fires share the precision-mismatch family but operate at different canonical-state-read layers. Naming the partial-absorption fingerprint as a distinct fire shape within the class clarifies that the verification-query precision class spans more than verification-query drafting itself. Anywhere canonical schema is being propagated by AI assistance (fresh-session verification queries, operational-copy persona sections, downstream framing prose, derivative artifacts), the precision class can fire when the propagation operates from session-memory rather than literal canonical-schema read.

The half-asymmetric absorption signature is empirically diagnostic. One canonical bullet caught; one canonical bullet missed. It's not random drift; it's pattern-completion bias against partial structural similarity. When a canonical schema has parallel structure (Hall Monitor embeds Teacher's Pet; Quarterback embeds New Kid), session-memory drafting may absorb one branch of the parallel and miss the other. Single-substrate verification pattern-matches past this because the substrate sees one branch present and assumes structural completeness; only literal canonical-schema read forces verification of both branches.

This is anti-pattern #10's canonical-source discipline operating at a layer that anti-pattern #10's original surface didn't anticipate. Anti-pattern #10 governs substantive-claim drafting against canonical record. Topic 34 extends the discipline to query-drafting and operational-copy text propagation; the precision class is anti-pattern #10's family rule applied at the verification-query and canonical-text-propagation layers respectively.

## What the class catches and what it costs

What single-substrate verification misses: a verification query drafted from session-memory of canonical state will pass single-substrate review by both the drafting substrate and the executing substrate, because both operate from the same calibration bias. Claude drafts the query against an inferred-or-composite taxonomy; Grok executes the query against the same inferred-or-composite taxonomy; both report PASS. The mismatch with literal canonical state only surfaces when an audit tests the query against canonical schema rather than against substrate session-memory.

Step 10 cross-substrate verification reported PASS twice during S009 corrective sub-stage 1 (per master-setup/lessons-learned.md line 549). Both passes were single-substrate-class verification: Claude + Grok inside the same operating session, sharing calibration bias. Gemini Audit A's third-leg independence (covered in [Post 20](0020-council-of-three-third-leg-fallibility.md)) was the audit mechanism that caught the verification-query precision fire. Without the third-leg audit, the precision-class fire would have shipped into operator-owned task lists, where the operator would have run verification queries that returned correctly-failing results against canonical state and concluded canonical state was broken when canonical state was actually correct and the queries themselves were the artifact.

What it costs: literal canonical-schema read before each verification-query draft and before each operational-copy text propagation. The cost is enforcement attention at every drafting site, not tooling enforcement. Verification-query drafting can't be tooling-checked because the canonical schema is itself textual content that requires reader interpretation; tooling can pattern-match for "all 6 Beatles names appear in the query" but can't pattern-match for "the query enumerates against the literal canonical role architecture." The class is reading-discipline work, structurally similar to Topic 33 (article-stacking discipline; covered in [Post 22](0022-topic-33-article-stacking-discipline.md)) and to anti-pattern #10's parent rule.

What it buys: verification queries and operational-copy text that match canonical state literally. Downstream operator verification work tests real canonical state, not approximated state. Council of Three triangulation rounds catch the precision-class fires that intra-substrate verification can't catch, with the structural cost of routing high-stakes canonical-state work through third-leg-independence audit. The two-for-two empirical record at S009 (per master-setup/lessons-learned.md line 555: Gemini audit catches drift that Claude plus Grok intra-substrate verification reports as zero across both corrective sub-stages) supports the audit mechanism's load-bearing role for precision-class detection.
