## The buffer pattern

The pattern is common enough that most codebases have several instances of it. A bounded-history buffer holds the last N entries of some event stream. When new events arrive, the old buffer plus new entries becomes the new buffer, capped at N. The idiomatic shape in most modern languages that support immutable-array patterns looks like this.

```
newBuf = [...oldBuf.slice(-N), ...newEntries]
```

The intuition is direct. Preserve the last N entries of the old buffer, then append the new ones. The cap is N. The append is on the tail. The oldest entries drop off as newer ones push in.

The intuition is correct when `newEntries` has one entry. The intuition is off-by-K when `newEntries` has K entries.

The math is elementary. `slice(-N)` returns at most N entries. Then K entries get appended. The result has between N and N+K entries depending on the old buffer's fill state. If the buffer was full at N, the result is N+K, not N. The cap-N intent is violated by K entries every fire.

For a strictly capped buffer at N with K entries added per fire, the correct slice is `slice(-(N-K))`.

The pattern is not exotic. Any event-stream buffer where a single logical event produces multiple entries (a bidirectional relationship event producing a directional record for each participant, a paired transaction producing debit and credit entries, an audit event producing both the raw event and its derived summary) faces this pattern. The buffer-slice discipline needs entry-count-per-fire accounting to enforce the cap correctly.

## The specific fire

The buffer where this hid is a relationship-history buffer in the framework's active empirical BUILD-mode arc. The framework's simulation subsystem models ordered-pair sentiment between two agents observing each other over time. When the two agents co-occupy the same location for a sustained interval, a "co-observed" event fires. The event is directional (one entry for agent A observing agent B, one entry for agent B observing agent A) because the ordered-pair sentiment scalars canonized at an earlier post are asymmetric. Every fire appends two entries.

The buffer's cap is 40 entries per the framework's canonical hypothesis. The append site used `slice(-39)`. The reasoning at authoring time was that the old buffer preserves 39 entries then the new ones append. The reasoning missed that "the new ones" plural meant two entries, so the buffer would settle at 41 when both terms were maximum. The bug was one character off from correct: `slice(-38)` was needed for two-entry appends.
