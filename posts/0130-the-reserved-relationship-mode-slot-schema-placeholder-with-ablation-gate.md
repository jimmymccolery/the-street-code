## The reserved slot

The framework's V1 architecture reserves one enumerated field per ordered pair on the Layer 2 consolidated pair-summary record:

`relationship_mode`. An enumerated field with five values: `COMMUNAL_SHARING`, `AUTHORITY_RANKING`, `EQUALITY_MATCHING`, `MARKET_PRICING`, and one sentinel value `UNDEFINED_OR_MIXED`. The four substantive enumerations correspond to the four Fiske Relational Models Theory primitives. The sentinel handles both the pre-assignment state (before the consolidation pass assigns a mode) and the mixed state (where the pair's interactions do not resolve to a single mode across the observed period).

The field ships as `UNDEFINED_OR_MIXED` for all ordered pairs at V1. The V1 game-logic does not branch on the field; the V1 game-logic proceeds against the existing decomposition (HEXACO honesty-humility trait plus attachment style plus Hawkins-Carrere-Gottman sentiment override, documented in posts 0115 and 0116 in this arc). The field is reserved schema for forward-compatibility; the V1 ships with the slot without operationalizing the mode-gate on decision-scoring.

The pattern is parallel to the schema-placeholder discipline documented in post 0128 in this arc. The framework's Layer 3 accumulator ships with three reserved metadata fields at V1 (`cultural_context_tag` for the WEIRD-defaults handling; `intersectional_position_tags[]` for the multi-axis identity representation; `identity_stage` for the Cross-Nigrescence staged-identity-development framework). Each field ships as reserved schema without operationalized V1 game-logic; the post-V1 work that would operationalize the fields depends on additional operator-side scholarship and audit-gate work; the fields reserve schema space so that the post-V1 operationalization does not require schema migration.

The `relationship_mode` field is the fourth reserved-slot the framework's architecture ships at V1. The rationale is different (the WEIRD-defaults handling reserves fields for cross-cultural-generalization work; the `relationship_mode` field reserves the slot for the Fiske Relational Models Theory operationalization pending the ablation gate); the discipline is the same (schema space reserved without operationalized V1 game-logic; forward-compatible; near-zero V1 cost; post-V1 operationalization possible without schema migration).

## Storage of record and runtime access surface

The framework's three-layer memory architecture (documented in post 0118 in this arc) has three plausible homes for a reserved `relationship_mode` field. The Layer 2 consolidated pair-summary is the correct home; Layer 1 (raw events) and Layer 3 (accumulator) are architecturally mis-placed for the specific field's semantics.

The Layer 2 pair-summary is the storage of record. The Layer 2 consolidated pair-summary record is the framework's compressed fixed-size deterministically-generated artifact for each ordered pair, regenerated at the yearly consolidation pass (documented in post 0116 in this arc for the pair summary's specific semantics and in post 0118 in this arc for the consolidation-pass mechanism). The `relationship_mode` field is one more bounded field on that artifact; the mode assignment happens at the consolidation pass as a bounded computation over the year's pair-relational events; the storage cost is one enumerated value per ordered pair per consolidation cycle.

The Layer 1 raw-events store is not the correct home. Layer 1 is bounded to the approximately six-to-twelve-game-month recency window per character (documented in post 0118 in this arc); the mode is a slow-moving property of the pair rather than a property of each event; tagging every raw event with the mode inflates the per-event pointer overhead that post 0122 in this arc budgets at approximately 50-100 bytes per event across the four typed indexes. The mode belongs at the pair-summary layer where the summary is regenerated at the consolidation pass, not at the raw-events layer where each event carries its own overhead.

The Layer 3 accumulator is not the correct home. Layer 3 is the per-character running affective state (valence, arousal, stress; constant per-character storage cost; documented in post 0128 in this arc for the accumulator schema plus the reserved metadata fields). The `relationship_mode` field is per-ordered-pair rather than per-character; the field is categorically mis-placed in the per-character accumulator; the mode belongs at the pair-summary layer where per-pair state lives.

The typed-index entity-view is the runtime access surface. The framework's multi-graph memory typed-index architecture (documented in post 0122 in this arc; the typed-index adaptation of MAGMA-class agentic-memory architectures with four indexes for semantic, temporal, causal, and entity views) already returns the pair summary via the entity index keyed by ordered pair. The mode read piggybacks on the already-performed entity-index lookup at zero additional lookups; the mode-gate would be a branch-and-sign-flip on scoring terms that already compute. The runtime cost of the mode read is a constant-time enumerated-value comparison per scored candidate action, well inside the per-frame budget the framework's cached-policies-with-slow-consolidation pattern (documented in post 0123 in this arc) reserves.

The composed placement (storage of record at Layer 2 pair-summary; runtime access surface at the typed-index entity-view) requires no new index, no fifth memory view, no Layer 1 change, no Layer 3 change, no migration penalty later. The placement is the minimum-honest architectural home for the reserved slot.

## The four-condition ablation gate

The V1.x operationalization (the mode-gate on decision-scoring plus the cached-policy integration under Path A's sophisticated active inference) is banked behind a four-condition ablation gate. The ablation gate composes the ablation mechanism specified during Council #1 (originally proposed by the framework's implementation-schema minimization substrate) with the tycoon-population setting specified during Council #1 (originally proposed by the framework's commercial-precedent breadth substrate); the composition produces the concrete evidence surface the operationalization commitment depends on.

Ablation Condition 1: mode-blind ablation. Run the consolidation-pass cached-policy regeneration with the `relationship_mode` field forced to `UNDEFINED_OR_MIXED` for all ordered pairs and compare the resulting cached policies against the same regeneration with assigned modes. If the cached policies (and therefore the NPC decisions the audit re-evaluates) are indistinguishable under the expected-free-energy re-scoring, the mode is inert and fails its justification. The framework's existing decomposition (HEXACO honesty-humility plus attachment plus sentiment override) already covers the variance; the mode does not do differential work; the operationalization is not justified.

Ablation Condition 2: mode-permutation ablation. Swap a pair's mode value (for example, from `MARKET_PRICING` to `COMMUNAL_SHARING`) holding the sentiment scalar and the event history fixed and verify that the same event sequence yields a different obligation-interpretation in decision-scoring. If the mode-gate does not actually gate (the decision-scoring output is unchanged across the swap), the mode is not doing the interpretive-grammar-selection work the theoretical framing assigns to it, and the operationalization is not justified.

Ablation Condition 3: sharpened sentiment-collinearity ablation. Confirm the `relationship_mode` field is not merely a coarse re-encoding of the sign or magnitude of the framework's existing pair-relational sentiment scalar (the ordered-pair sentiment scalar in the range from negative one to positive one, documented in post 0116 in this arc). This condition is the load-bearing one; the sharpening is what the Council #1 orchestrator layer surfaced (documented in post 0129 in this arc). The Zakharin and Bates 2023 paper (PLOS ONE 18(6):e0287391; independently verified at the primary source) finds that the four-factor Fiske Relational Models Theory structure requires a general factor to fit, and Study 4 characterizes the general factor as associated with relationship closeness. Closeness is the construct-neighbor of the framework's sentiment scalar; the primary-source empirical evidence Substrate A cited as proof of Fiske Relational Models Theory's non-redundancy contains the specific reason to expect partial collinearity between the mode assignment and the sentiment scalar. The sharpened ablation quantifies mode-variance surviving after partialling out the closeness-linked general factor from the sentiment scalar. That residual is the actual justification surface for the enum. The ablation must not merely check whether mode is distinguishable from sentiment; the ablation must quantify how much of the mode's variance is orthogonal to the closeness-linked general factor. If the residual is negligible, the mode is not adding differential-variance work that the existing decomposition does not already cover, and the operationalization is not justified.

Ablation Condition 4: WEIRD-defaults disclosure check. Because the Fiske Relational Models Theory framework's empirical validation is WEIRD-sample-based (Zakharin and Bates 2023's samples are British; the broader Fiske Relational Models Theory validation literature has similar WEIRD-sample skew), the framework's discipline requires the mode assignment to be gated behind the `cultural_context_tag` schema placeholder rather than defaulting all ordered pairs to a Western relational-mode distribution. The condition parallels the schema-placeholder discipline documented in post 0128 in this arc for the WEIRD-defaults handling across the framework's broader architecture (HEXACO trait validation; attachment style validation; sentiment-override validation; the Layer 3 accumulator's calibration parameters). If the mode-assignment heuristic silently defaults WEIRD without the `cultural_context_tag` gate, the operationalization inherits the WEIRD-default failure mode the framework's broader discipline exists to prevent; the disclosure check catches the failure mode at the operationalization step.

The four conditions are conjunctive. All four must pass for the operationalization to proceed. If Conditions 1, 2, or 3 fail, the mode is inert or redundant or collinear and the operationalization is not justified; the reserved slot is retired at near-zero sunk cost (the V1 shipped the slot as `UNDEFINED_OR_MIXED` for all pairs; retiring the slot requires removing the field from the pair-summary schema, which is bounded work). If Condition 4 fails, the disclosure gate is not honored; the operationalization inherits the WEIRD-default failure mode; the operationalization is deferred pending the operator-side scholarship that would support the WEIRD-defaults-aware mode assignment.

The composed gate runs the ablation conditions against the framework's specific tycoon-population scenarios: staff-loyalty relationships across the multi-decade employment arc; customer-patronage relationships across recurring customer visits; league-and-competitor-dynamics relationships across the framework's league-play mechanics. The setting matters because implementation-collinearity is a population-specific property (documented in post 0129 in this arc): construct-level distinctness in Zakharin and Bates 2023's British general population is not the same object as implementation-level orthogonality in the framework's specific tycoon-domain NPC population. The tycoon-population scenarios are the setting where the ablation-gate's evidence surface actually lives.

## Cost arithmetic

The reserved-slot addition at V1 costs approximately 2-5 hours of BUILD-side work: schema migration to add the enumerated field to the pair-summary record; consolidation-pass extension to preserve the field across regeneration (the field ships as `UNDEFINED_OR_MIXED` without an assignment heuristic; the extension is bounded); pair-summary regeneration test-run validation. The cost sits within the framework's V1 slot-reservation budget precedent (the three reserved fields documented in post 0128 in this arc ship at similar order-of-magnitude cost); the reservation is forward-compatible; the reservation carries no verdict risk.

The V1.x operationalization cost estimate is approximately 36-72 hours of BUILD-side work: schema migration to expand the enumeration's operationalization; consolidation-pass extension to assign the mode via the mode-assignment heuristic; pair-summary regeneration extension to include the mode as an input to the summary; cached-policy regeneration extension to make the mode one more input to the Tier-2 cached-policy bake (per the cached-policies-with-slow-consolidation pattern documented in post 0123 in this arc); decision-scoring mode-gate that selects which existing scoring terms fire and with what sign based on the assigned mode; audit-layer extension to verify the mode-discipline compliance under the audit-layer apparatus documented in post 0119 in this arc; cross-cultural disclosure obligation extension per the four-condition ablation gate's WEIRD-defaults disclosure check; multi-game-year test-run validation across the tycoon-population scenarios the ablation gate specifies.

The 36-72 hour estimate brackets the framework's own 33-88 hour estimate for the structurally-comparable Wireduan generosity-ledger relational primitive documented in post 0126 in this arc. The framework banked the generosity-ledger mechanic to Steam V1.0 rather than iOS V1 because even Path A's extended budget (270-410 hours per the Path A operator ratification documented in post 0119 in this arc) does not fit comfortably with the addition. The same arithmetic applies to the `relationship_mode` operationalization: 36-72 hours is approximately 9-27% of the entire remaining iOS V1 budget against a fixed 2026-10-31 ship date. The operationalization is banked V1.x scope; the ablation gate specifies exactly what evidence unlocks the operationalization at V1.x scope.

## What the reserved slot enables and what it does not

The reserved slot enables the framework's future architectural decisions to include the mode-gate operationalization without schema migration. If the ablation gate's four conditions all pass, the operationalization proceeds at V1.x with the schema already in place; the operationalization adds the mode-assignment heuristic and the mode-gate on decision-scoring and the cached-policy integration to the existing schema slot rather than requiring a new schema. The schema-placeholder discipline (per post 0128 in this arc) is what makes the V1.x operationalization operationally cheap conditional on the gate.

The reserved slot does not commit the framework to operationalize the mode-gate at V1.x. The commitment is conditional on the ablation gate's four conditions. If any condition fails, the reserved slot is retired; the retirement is bounded work (the field ships as `UNDEFINED_OR_MIXED` for all pairs at V1, so no game-logic depends on the field; removing the field from the pair-summary schema is a bounded schema migration). The commitment is to the reservation now and to the ablation gate later, not to the eventual operationalization.

The reserved slot does not solve the implementation-collinearity question the framework's Council #1 orchestrator layer identified. The slot reserves schema space; the collinearity question is empirical; the ablation gate is what produces the evidence. The primary source Substrate A cited as proof of Fiske Relational Models Theory's non-redundancy (Zakharin and Bates 2023) contains the specific empirical seed of the collinearity worry (the closeness-linked general factor Study 4 characterizes). The ablation gate's Condition 3 is the sharpened test that quantifies the residual after partialling out the closeness factor; the residual is the justification surface for the enum; the residual is what the ablation gate produces.

The reserved slot does not extend the framework's disclosure discipline to the mode-assignment heuristic. The mode-assignment heuristic is not specified in this post (the heuristic depends on the ablation gate's Condition 4 disclosure gate; the heuristic depends on the framework's `cultural_context_tag` schema placeholder from post 0128 in this arc; the heuristic depends on the framework's broader cross-cultural-generalization work that the schema-placeholder approach reserves space for). The mode-assignment heuristic is V1.x operationalization scope; the reserved slot ships without the heuristic at V1.
