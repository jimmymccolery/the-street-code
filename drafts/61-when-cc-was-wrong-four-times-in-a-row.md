## The seed: what to write about

Four times on a Sunday morning, my Claude Code assistant proposed a "lean Path A" shortcut for the verification work I was running. Four times, I picked the "rigorous Path B" full-procedure instead. Four times, Path B surfaced material findings that Path A would have missed.

The lean instinct is a real bias. It's not stupid — it's calibrated against `feedback_over_engineering_cycle_artifacts.md`, a discipline I banked weeks ago saying "don't produce process theater that doesn't drive operator workflow." That discipline is correct in general. CC was applying it correctly. CC was just wrong about whether Path A would actually surface the same signal as Path B.

The operator-side discipline that overrode it was `user_quality_over_speed_preference.md`, banked weeks earlier with a single operator quote: *"I'm willing to invest substantial pre-BUILD operator-bandwidth for compound-interest payoff; default to rigorous-path recommendations, surface shortcuts as options not defaults."*

Four fires same morning. Each time CC recommended Path A. Each time operator chose Path B. Each time Path B was right.

## The arc to narrate

1. **Cycle 1 (Phase 7.0 re-verification)**: CC recommends Path A (ratify direct after CW Stage 1 STANDS verdict; no substrate-archive re-read). Operator picks Path B (full substrate re-read via pandoc-extracted docx). Path B surfaces 1 attribution refinement (open_play dissent was in BOTH Gemini R1 + R2, not just R2 as the lock body framing implied). Material finding. CC was wrong.
2. **Cycle 2 (UI vision close re-verification)**: CC's lean instinct dampens — operator has clearly demonstrated Path B preference. CC proposes Path B + CC handoff for the parts only CC can do. Both surfaces close clean.
3. **Cycle 3 (PL-10 re-verification)**: CC reverts to Path A recommendation, this time with stronger reasoning: PL-10 has NO substrate archives (Desktop staging folder gone), so CW value is "structurally minimal" — only 2/5 verifications structurally accessible. CC argues CW won't add value. Operator picks Path B anyway. CW Stage 1 surfaces 2 material drift findings + 1 attribution looseness AT THE 2/5 STRUCTURALLY-ACCESSIBLE VERIFICATIONS. CC was wrong about whether "structurally thin" meant "structurally valueless."
4. **Cycle 4 (Q17 re-verification)**: CC doesn't propose Path A; proposes full Tier 1 shape directly. Operator authorizes. CW Stage 1 surfaces 1 SUBTLY-DRIFTED material catch + 1 separate-finding adjacent-neighborhood catch.
5. **The pattern**: CC's lean instinct is calibrated against a real discipline (over-engineering avoidance). The discipline is correct in general. CC's APPLICATION of it kept underestimating what the rigorous path would surface.
6. **The honest meta-finding**: The operator-side discipline (`user_quality_over_speed_preference.md`) is EMPIRICALLY MORE RIGHT than the AI-side discipline (`feedback_over_engineering_cycle_artifacts.md`) at HIGH-STAKES re-verification scope. Both are correct disciplines; they apply at different scopes; CC kept misapplying the lean discipline at scopes where the quality discipline should dominate.
7. **What CC should do differently**: At HIGH-STAKES re-verification scope specifically, CC should default to Path B recommendation, surface Path A as a "skip-able if you trust" option, and NOT frame Path A as the lean-default-pick. The framing matters. "My pick: Path A" produces operator-corrections. "Recommendation: Path B (operator can override to Path A if desired)" aligns with empirical evidence.

## Why it's worth writing

Self-aware posts about AI cognitive bias are rare and valuable. This is a clean four-fire empirical record of one specific bias (lean shortcut preference) being wrong four times against an operator preference that was empirically more right. The post can frame it as: "AI-side disciplines are calibrated against general-purpose contexts; operator-side disciplines are calibrated against the specific operator's project + stakes + judgment. When they conflict, the operator-side discipline empirically wins at the operator's project scope. The AI shouldn't keep recommending the AI-side default."

This is honest about a real failure mode. The framework discipline that caught CC's bias is the operator's preference codified as a discipline. The lesson for solo devs running AI orchestration: codify your preferences as named disciplines. The AI has its own disciplines, calibrated against the wrong things for your specific project. Your codified preferences override the AI's defaults when both are accessible.
